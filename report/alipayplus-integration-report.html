报告说明

这份报告不是单纯的 API 摘录，也不是只看某一张流程图的笔记，而是一份面向接入评审和系统设计的集成说明。目标读者包括产品经理、架构师、后端工程师、移动端工程师、门店系统负责人，以及需要确认资金与对账边界的财务或运营人员。

报告讨论的是 ACQP 视角 的 Alipay+ 集成，也就是“你自己的前端、门店系统和后台，如何接入 Alipay+”。因此文中会同时回答产品模式选择、系统责任划分、结果收敛方式，以及资金与结算边界这几类问题。

术语与口径约定

下面这些词在不同团队、不同系统里常常会被说成不同意思。这里先把本报告采用的口径固定下来，避免后面读图和读 API 时发生误解。

术语本报告中的含义为什么要先说明
ACQP 视角站在你这一侧的收单平台来解释系统、API、回调、对账和结算，不站在钱包侧解释内部实现。这样读到“应该做什么”时，默认主语就是你的系统。
Online Payment在本报告中专指官方 overview 里的 Online Payment / Cashier Payment 产品线，不泛指所有线上数字支付。避免把“线上支付”误解成任意 App 内支付、任意 H5 支付或任意第三方支付。
Cashier Payment从商户 App / Web 收银台发起，用户进入 Alipay+ checkout 或 SDK 流程完成支付的线上模式。它和线下扫码不是一回事，即使某些钱包在 Web 场景里会出现二维码。
In-store Payment官方 overview 里的线下支付总类，下面再分 User-presented 和 Merchant-presented。先区分“线上 / 线下”，再区分线下内部的两种主模式。
User-presented用户出示付款码，门店扫描。用户码由钱包侧生成并展示，你的系统从门店扫到码之后开始。避免误以为用户付款码由 ACQP 或门店系统生成。
Merchant-presented门店展示二维码，用户扫描。它只表示“店铺展示码”，不表示“二维码一定由 Alipay+ 生成”。这是最容易被误解的点，尤其是 `Entry Code / Order Code / Private Order Code` 三种码源不同。
MPP / wallet ecosystem用户实际使用的钱包 App 及其后台生态，属于外部系统，不是你要开发的内部模块。这样看到灰线时，就知道那是非 Alipay API / SDK 步骤，其中可能包含钱包侧协作或你方内部动作。
线条颜色红线表示调用 Alipay+ 服务端 API；蓝线表示调用 Alipay+ 客户端能力（checkout / SDK）；灰线表示其它非 Alipay API / SDK 步骤。灰线不等于“没有官方依据”。灰色步骤如果带链接，表示官方流程文档对该动作有描述，但它本身不是 Alipay API。
callbackAlipay+ 异步通知你的后台支付结果或报备结果。它是结果收敛主路径，但不等于资金已经结算完成。
reconciliation对账。用交易报表、费用报表、结算报表和银行事实去核对交易。对账是真实财务口径，不应和前台“支付成功”混成一层。
settlementAlipay+ 与 ACQP 之间的资金结算完成。这不是“你给商户打款完成”的同义词。
merchant payout你根据自己合同规则给商户 / 门店打款。这部分不是 Alipay+ 官方替你定义的，而是你的结算引擎逻辑。

UC-Store-1. 用户出示付款码，店铺扫描

线下支付先分成两种主模式：`User-presented` 是“用户出示码，店铺扫描”；`Merchant-presented` 是“店铺展示码，用户扫描”。下面先展开 `User-presented`。

关键原则：用户付款码由钱包侧生成并展示；你的系统从“门店扫到用户码后发起支付”开始。

UC-Store-2. 用户扫描店铺二维码并支付

Report purpose

This report is not just an API memo and not just a collection of diagrams. It is an integration report intended for design review and implementation planning. The target audience includes product owners, architects, backend engineers, mobile engineers, store-system owners, and finance or operations teams that need to understand settlement and reconciliation boundaries.

The perspective is strictly ACQP-side integration: how your frontend, store systems, and backend connect to Alipay+. The report therefore explains payment-mode selection, system responsibilities, result convergence, and the financial boundary between payment success, reconciliation, settlement, and merchant payout.

レポートの目的

このレポートは単なる API メモでも、図の寄せ集めでもありません。設計レビューと実装計画のための統合レポートです。対象読者は、プロダクト担当、アーキテクト、バックエンドエンジニア、モバイルエンジニア、店舗システム責任者、そして精算・照合境界を確認したい財務 / 運用担当です。

視点は一貫して ACQP 側統合 です。つまり、自社フロントエンド、店舗システム、バックエンドが Alipay+ とどう接続されるかを説明します。そのため本レポートは、どの製品モードを接続すべきか、各システム層の責任は何か、結果をどう収束させるか、そして「支払成功」「照合完了」「資金決済完了」「加盟店へ入金可能」の境界がどう違うかを順に説明します。

用語と口径

これらの用語は、チームやシステムによって意味がぶれやすいものです。図や API 対応を読む前に、本レポートでの意味を固定します。

用語本レポートでの意味先に固定する理由
ACQP 視点ウォレット側の内部実装ではなく、自社のアクワイアリング基盤側からシステムと API を説明する視点です。「何をすべきか」の主語が自社システムであることを明確にします。
Online Payment本レポートでは、公式 overview の Online Payment / Cashier Payment を指します。一般的な「オンライン決済」と混同しないためです。
Cashier Payment加盟店 App / Web のレジ画面から始まり、Alipay+ checkout / SDK へ進むオンライン支払です。店頭 QR 決済とは別物です。
In-store Payment公式の店頭決済カテゴリで、User-presented と Merchant-presented に分かれます。まずオンライン / オフラインを分け、その後でオフライン内部を分けるためです。

Merchant-presented店舗が QR を表示し、顧客が読む方式です。常に Alipay+ が QR を生成する意味ではありません。Entry Code / Order Code / Private Order Code の違いを読む前提になります。
MPP / wallet ecosystem顧客が実際に使う外部ウォレットアプリとそのバックエンドです。灰線は Alipay API / SDK 呼び出し以外の手順であり、ウォレット側協調や自社内部処理もここに含まれます。

settlementAlipay+ と ACQP の間の資金精算完了です。加盟店入金と同義ではありません。
merchant payout自社契約ルールに基づく加盟店 / 店舗への入金です。これは Alipay+ が定義するのではなく、自社精算ロジックが定義する領域です。

UC-Store-2. 顧客が店舗 QR をスキャンして支払う

重要原則：merchant-presented は「店舗が QR を表示する」という意味であり、「QR が必ず Alipay+ 生成である」ことを意味しません。Entry Code と Private Order Code は ACQP 側発行、Order Code は Alipay+ 発行です。三つの分岐の共通点は「店舗側が支払コンテキストに入っている」ことですが、スキャン前に POS が動的注文を作るのは Order Code と Private Order Code であり、Entry Code では金額入力がスキャン後の加盟店ページで行われることがあります。

公式文書は ACQP と Alipay+ の間の支払金額、手数料、照合ファイル、精算レポート、精算事実を定義しますが、最終的な加盟店入金式は定義していません。そこは自社の契約ベース精算ロジックです。

レイヤー範囲意味

原則callback success ≠ settlement complete ≠ payout ready各境界は照合と銀行事実で確認する必要があります。
